# Phase 05 Plan 01: Reverse Matching Implementation

**One-liner:** Implemented candidate-to-JD reverse matching (MATCH-04) with backend SSE endpoint, skill-overlap scoring, MatchSession schema migration, and full frontend integration from CandidateDetail button to Matching page.

## What Was Built

Reverse matching allows HR to find matching JDs for a candidate from the CandidateDetail page. The implementation reuses the existing Matching page infrastructure and SSE streaming pattern.

### Backend (REST + SSE)

- **Migration:** MatchSession.jd_id changed to nullable, candidate_id column added, results_json Text column added. Alembic batch mode enabled for SQLite compatibility.
- **Endpoint:** `POST /recruit/hitl/reverse-stream` accepts `{ candidate_id: int }` and streams SSE events.
- **Scoring:** Simple skill-overlap algorithm (intersection/union ratio) computed in-memory against all active JDs -- no LangGraph agent required.
- **Session tracking:** MatchSession created with candidate_id, results_json stored as serialized JSON for export use.
- **Forward compat:** Existing `_stream_hitl_recruitment` updated to include `session_id` in `hitl_pause` events (needed by Plan 05-03 export).

### Frontend

- **Type:** `HITLPausePayload` extended with optional `session_id`.
- **API:** `startReverseMatchingStream(candidateId, signal)` function added to matching API.
- **Button:** CandidateDetail page now shows a "反向匹配" card with a button navigating to `/matching?candidateId=X`.
- **Hook:** `useMatchingSSE` extended with `startReverse(candidateId)` method and `sessionId` state. Tracks `session_id` from SSE hitl_pause events.
- **Page:** Matching.tsx detects `candidateId` query param, calls `startReverse` on mount. IDLE/DONE messages contextualized for reverse mode.

## Deviations from Plan

**1. [Rule 3 - Blocking] Alembic batch mode required for SQLite**

- **Found during:** Task 1 (migration application)
- **Issue:** SQLite does not support `ALTER TABLE ... ALTER COLUMN ... DROP NOT NULL`. The initial autogenerated migration used `op.alter_column()` which executes raw ALTER COLUMN DDL, causing `sqlite3.OperationalError`.
- **Fix:** Added `render_as_batch=True` to Alembic env.py `context.configure()`. Regenerated migration with `with op.batch_alter_table()` which recreates the table transparently for SQLite.
- **Files modified:** `backend/db/migrations/env.py`
- **Commit:** `b6fed1f`

**2. [Rule 1 - Bug] Potential UnboundLocalError for match_session reference**

- **Found during:** Task 1 (Part C - adding session_id to hitl_pause)
- **Issue:** The `match_session` variable was defined inside `if parsed_results:` block but referenced outside it in the `yield _sse_event("hitl_pause", ...)` call. If `parsed_results` was empty, this would raise `UnboundLocalError`.
- **Fix:** Moved `match_session = None` before the `if parsed_results:` block; used `match_session.id if match_session else None` in the event payload.
- **Files modified:** `api/server.py`
- **Commit:** `b6fed1f`
