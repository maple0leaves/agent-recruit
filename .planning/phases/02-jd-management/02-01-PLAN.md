<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> API | Untrusted HTTP requests cross into the API layer. All JD endpoints authenticated via Depends(get_current_user). |
| API -> Database | SQLAlchemy ORM queries. Input data flow from Pydantic models to ORM inserts/updates. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-JD-01 | Tampering | POST/PUT /jd (mass assignment) | mitigate | Pydantic JDSchema defines exact allowed fields. No direct ORM model_dump() without schema filtering. Status changes only via dedicated PATCH /{id}/status endpoint. |
| T-JD-02 | Information Disclosure | GET /jd (list endpoint) | mitigate | SQLAlchemy ilike() uses parameterized queries; not vulnerable to SQL injection. Keyword filter uses ORM-level escaping. |
| T-JD-03 | Elevation of Privilege | All JD endpoints | mitigate | D-14: any logged-in user can create/edit any JD — no ownership check needed. All endpoints require Depends(get_current_user). No role-based restrictions on JD. |
| T-JD-04 | Denial of Service | GET /jd list | accept | No PII exposed in JD data. Page size limited to 100 (Query ge=1, le=100). SQLite COUNT on filtered queries is acceptable for <100K rows. |
| T-JD-05 | Tampering | PATCH /jd/{id}/status | mitigate | D-12 state machine strictly enforced. Invalid transitions return 400 with descriptive error. Only three valid transitions allowed. |
</threat_model>
