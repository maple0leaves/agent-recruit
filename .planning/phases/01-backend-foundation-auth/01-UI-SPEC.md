# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Backend Foundation & Authentication.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Overview

Phase 1 delivers the complete frontend application shell: login page, protected sidebar+header layout, and route placeholders. No data features are built yet — this is purely the auth-enabled scaffold that all subsequent phases fill with content.

**Deliverables:**
- `/login` — Public login page with username+password form
- `/` — Redirects to `/dashboard`
- `/dashboard`, `/jd`, `/candidates`, `/matching` — Protected placeholder pages behind auth guard
- `Layout.tsx` — Sidebar + topbar shell wrapping protected routes
- `ProtectedRoute.tsx` — Auth guard that redirects unauthenticated users to `/login`

**Status after Phase 1:** HR can log in, see a navigation sidebar, and visit placeholder pages. Pages show "coming soon" state. Session persists across refresh.

## Component Inventory & States

### 1. Login Page (`/login`)

**Layout:** Centered card on Slate-50 background. Full viewport height.

| Element | Component | States | Details |
|---------|-----------|--------|---------|
| Branding | Heading (Display) | - | "HR 智能招聘系统" at top of card |
| Subtitle | Body text | - | Small text below brand: "请登录以继续" |
| Username | shadcn Input | default, focus, error | Placeholder: "用户名" |
| Password | shadcn Input (type=password) | default, focus, error | Placeholder: "密码" |
| Login button | shadcn Button (primary) | default, hover, active, loading, disabled | Label: "登录"; shows spinner while submitting |
| Error message | Body text (destructive color) | visible / hidden | Shows server error or validation error; placed above submit button |
| Card wrapper | shadcn Card | - | CardHeader: brand + title; CardContent: form |

**Loading state:** Button shows a spinner icon (Lucide `Loader2`) and text "登录中..." while the login API call is in-flight. Input fields remain enabled so user can correct credentials.

**Error state — wrong credentials:** The form stays visible. Button returns to "登录" state. An error message appears above the button in destructive red: "用户名或密码错误". The password field is cleared.

**Error state — network failure:** Same as above but message: "网络连接失败，请检查网络后重试". Both fields keep their values.

**Validation:**
- Username: Required, min 2 characters. Empty submit shows field-level error: "请输入用户名"
- Password: Required, min 1 character. Empty submit shows field-level error: "请输入密码"
- Validation runs on submit only (not on keystroke — to avoid premature errors)

**CSS:** Tailwind: `flex min-h-screen items-center justify-center bg-slate-50`

### 4. ProtectedRoute Guard

| State | Behavior | UI |
|-------|----------|-----|
| Loading | Checking auth status | Centered `Skeleton` or spinner |
| Authenticated | Render `<Outlet />` | Normal layout with sidebar |
| Unauthenticated | Redirect to `/login` | `Navigate` replace (no flash of login page) |
| Token refresh failure | Redirect to `/login` | Clean session termination |

**Implementation pattern:**
- On app mount, call `GET /auth/me` with `withCredentials: true`
- If 200: store user info in auth context (Zustand store), render app
- If 401: clear auth state, redirect to `/login`
- Use TanStack Query's `useQuery` for `/auth/me` with staleTime: 0 (always refetch on mount)

## Interaction Contract

### Login Flow

1. User navigates to any protected route → `ProtectedRoute` checks auth → no session → redirect to `/login?redirect={path}`
2. User enters credentials → clicks "登录"
3. Button enters loading state, form inputs remain editable
4. **On success:** Store access token in memory (Zustand), navigate to `redirect` query param or `/dashboard`
5. **On 401:** Show error message "用户名或密码错误", clear password field, button returns to default
6. **On network error:** Show error message "网络连接失败，请检查网络后重试", fields keep values

### Logout Flow

1. User clicks "退出登录" in sidebar
2. POST `/auth/logout` (cookie cleared server-side)
3. Clear in-memory auth state (Zustand)
4. Navigate to `/login`
5. No confirmation dialog needed for Phase 1 (no destructive data loss risk)

### Session Refresh Flow

1. User refreshes page at any protected route
2. `ProtectedRoute` fires `GET /auth/me` with `withCredentials`
3. Server reads HTTP-only refresh token cookie, validates, returns user data
4. On success: render app normally
5. On 401: redirect to `/login`
6. TanStack Query handles caching — set `staleTime: 0` to always refetch on mount

### Navigation Interaction

- Clicking a sidebar nav item navigates to the route
- The active item receives the "active" visual state (blue accent left border + tinted background)
- Clicking the same active item does nothing (no re-navigation)
- Browser back/forward buttons work normally (React Router handles)
