    ```typescript
    const STATUS_LABELS: Record<string, string> = {
      new: "新入库",
      screening: "筛选中",
      interview: "面试",
      hired: "已录用",
      rejected: "不合适",
    };
    ```

    **State variables:**
    - `const [selectedStatus, setSelectedStatus] = useState<CandidateStatus | null>(null)` -- for confirmation dialog
    - `const [statusNote, setStatusNote] = useState("")`
    - `const [dialogOpen, setDialogOpen] = useState(false)`
    - `const statusMutation = useUpdateCandidateStatus()`

    **Layout:**
    - Current status displayed using PipelineStatusBadge
    - Title: "流程状态" with pipeline visualization showing the flow: new -> screening -> interview -> hired/rejected
    - Available next transitions shown as clickable buttons:
      - For each valid next status in VALID_TRANSITIONS[currentStatus]:
        - `<Button variant="outline" onClick={() => { setSelectedStatus(nextStatus); setDialogOpen(true); }}>`
        - Button label: STATUS_LABELS[nextStatus]
    - If no valid transitions (terminal state D-15), show muted text: "已终态，不可变更" (D-15)
    - Status history: show current status_note (if any) and dates

    **Status change dialog:**
    ```typescript
    <Dialog open={dialogOpen} onOpenChange={setDialogOpen}>
      <DialogContent>
        <DialogHeader>
          <DialogTitle>确认状态变更</DialogTitle>
          <DialogDescription>
            将候选人从「{STATUS_LABELS[currentStatus]}」变更为「{STATUS_LABELS[selectedStatus]}」
          </DialogDescription>
        </DialogHeader>
        <div className="space-y-4 py-4">
          <div className="space-y-2">
            <Label>变更备注 <span className="text-red-500">*</span></Label>
            <Textarea
              placeholder="请填写状态变更原因..."
              value={statusNote}
              onChange={(e) => setStatusNote(e.target.value)}
              rows={3}
            />
            {statusNote.trim() === "" && <p className="text-xs text-red-500">备注为必填项</p>}
          </div>
          <div className="flex justify-end gap-3">
            <Button variant="outline" onClick={() => setDialogOpen(false)}>取消</Button>
            <Button
              onClick={async () => {
                if (!statusNote.trim()) return;
                try {
                  await statusMutation.mutateAsync({
                    id: candidateId,
                    status: selectedStatus!,
                    status_note: statusNote.trim(),
                  });
                  setDialogOpen(false);
                  setStatusNote("");
                  onStatusChanged?.();
                } catch (err) {
                  // Error handled by mutation
                }
              }}
              disabled={!statusNote.trim() || statusMutation.isPending}
            >
              {statusMutation.isPending ? "保存中..." : "确认变更"}
            </Button>
          </div>
        </div>
      </DialogContent>
    </Dialog>
    ```

    **Error handling:**
    Show error in the dialog if mutation fails (e.g., invalid transition from backend). Add state `const [error, setError] = useState<string | null>(null)` and display in dialog.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> API (detail) | Candidate ID in URL/state, candidate data displayed from API response. |
| Browser -> API (status change) | Status and status_note sent via PATCH. Dialog requires non-empty note before enabling submit button. |
| Inline edit form -> API | Updated candidate fields sent via PUT. Client-side text inputs, server-side validation via Pydantic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-DTL-01 | Tampering | Inline edit form | mitigate | Frontend form collects updated fields, PUT sends to server. Backend Pydantic validates field types and lengths. CandidateUpdate schema limits which fields can be changed (no status, no parsed_at). |
| T-DTL-02 | Tampering | Status change dialog | mitigate | Dialog requires non-empty status_note before enabling confirm. Backend validates status_note non-empty and checks valid transitions. Terminal states have no transition buttons. |
| T-DTL-03 | Information Disclosure | Detail page display | accept | Candidate data displayed in browser. All candidate endpoints require auth. No sensitive data beyond what HR already has access to. |
</threat_model>
