# HelloJobs 最小全栈交付设计

## 目标

`hellojobs` 当前阶段的目标，不是先补齐所有工程化配置，而是先完成一个“前后端真实联动、具备完整业务闭环”的最小全栈项目，并以 `PostgreSQL` 作为正式数据库方案。

## 设计原则

- 优先业务闭环，不优先工程化堆料
- 保持现有 `React + FastAPI + SQLAlchemy + LangGraph` 主架构不变
- 以可演示、可运行、可说明白为当前验收标准
- `Docker`、`CI`、`Redis` 等能力可以后置

## 技术栈总览

### 后端

| 类别 | 技术 |
| ---- | ---- |
| Web 框架 | FastAPI |
| 运行时 | Python 3.11+ / Uvicorn |
| ORM | SQLAlchemy Async |
| 数据库迁移 | Alembic |
| 数据库 | PostgreSQL（`asyncpg`） |
| 认证 | JWT + bcrypt |
| AI/Agent 编排 | LangChain + LangGraph |
| 向量检索 | FAISS / ChromaDB |
| 简历解析 | PyMuPDF（PDF）+ python-docx（DOCX） |

### 前端

| 类别 | 技术 |
| ---- | ---- |
| 框架 | React 19 + TypeScript |
| 构建 | Vite 8 |
| 样式 | Tailwind CSS 4 + shadcn/ui |
| 路由 | React Router 7 |
| 状态管理 | Zustand |
| 数据请求 | TanStack React Query + Axios |
| 实时通信 | SSE |

### 测试

| 类别 | 技术 |
| ---- | ---- |
| 后端测试 | Pytest（async） |
| 前端测试 | Vitest |

## 最小交付标准

### 1. 前端页面闭环

至少需要打磨完整以下页面：

- 登录页
- 仪表盘 / 列表页
- JD 管理页
- 候选人管理页
- 匹配结果页

重点不在于页面数量，而在于这些页面必须真实调用后端接口，能够完成实际业务操作，而不是静态展示或假数据页面。

### 2. 后端接口闭环

后端至少需要稳定提供以下核心能力：

- 登录认证
- JD 增删改查
- 候选人上传 / 编辑 / 状态流转
- 发起匹配 / 查看匹配结果
- 仪表盘统计

后端不能只是“有路由文件”，而必须能真正支撑前端完成完整流程。

### 3. 数据库持久化

系统必须能够基于 `PostgreSQL` 持久化保存核心业务数据，至少包括：

- 用户
- JD
- 候选人
- 匹配记录 / 会话

对于 `hellojobs` 来说，数据库不只是“能存数据”即可，而是需要具备更接近真实业务系统的持久化能力。因此当前阶段以 `PostgreSQL` 作为正式数据库方案。

### 4. 文件上传真实打通

由于 `hellojobs` 是招聘系统，简历上传是关键业务能力，因此至少要做到：

- 支持上传 `PDF / DOCX`
- 后端能够解析或至少保存文件
- 前端能够看到上传结果
- 候选人记录和简历文件存在关联

这一点如果做扎实，项目会明显更像真实系统，而不是普通 demo。

### 5. 核心主流程跑通

项目至少应能完整演示以下链路：

`登录 -> 创建 JD -> 上传候选人 -> 发起匹配 -> 查看结果`

这是判断项目是否具备“全栈完成度”的关键标准，因为它证明做出来的是一个系统，而不是零散页面和孤立接口。

### 6. README 清晰可用

`README.md` 至少需要说明清楚：

- 项目是做什么的
- 技术栈
- 如何安装依赖
- 如何配置 `PostgreSQL`
- 如何启动前后端
- 环境变量如何配置
- 核心功能有哪些

如果别人看完 `README.md` 仍然不知道怎么运行项目，那么项目完成度会明显下降。

## 当前阶段不作为硬门槛的内容

以下内容属于加分项，可以后置：

- `Docker`
- `CI`
- `Redis`
- `Celery`
- `Sentry`
- 多角色权限
- 完整监控体系

## 推荐实现顺序

1. 先验证并修复主流程相关接口与测试
2. 再打磨前端关键页面交互与错误态
3. 补强 `README.md` 的最小启动路径和 PostgreSQL 配置说明
4. 最后再考虑补充前端最小测试

## 设计结论

`hellojobs` 当前最重要的目标，是先成为一个“基于 PostgreSQL、能运行、能演示、能说明白”的最小全栈项目，而不是一开始就做成工程化最完整的产品。
